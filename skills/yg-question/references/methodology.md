# 文档调查方法论

本文档详细说明 `yg-question` 技能的调查方法论，确保每个回答都有据可依。

---

## SEAR 调查框架

### 核心原则

**SEAR** 是文档调查的基础框架：

| 要素 | 说明 | 要求 |
|------|------|------|
| **S**ource | 来源 | 每个结论必须标注文档出处 |
| **E**vidence | 证据 | 引用原文内容作为证据 |
| **A**nalysis | 分析 | 基于证据进行分析推理 |
| **R**esult | 结论 | 得出明确结论 |

### 应用示例

```
用户问题：订单审批需要多长时间？

S - 来源：PRD-订单管理.md
E - 证据：原文"审批超时时间为48小时"
A - 分析：文档定义了审批时限，未说明特殊情况处理
R - 结论：正常审批时限为48小时，超时处理机制未定义
```

## 问题分解技术

### 分解原则

将复杂问题分解为可独立调查的原子问题：

1. **单一性**：每个子问题只询问一个事实
2. **可验证性**：每个子问题都能在文档中找到答案或确认缺失
3. **完整性**：子问题集合覆盖原问题的所有方面

### 分解模板

```
原问题："{用户问题}"

分解：
├── Q1: {能力确认} - 系统是否支持...？
├── Q2: {数据支撑} - 相关数据存储在哪些表？
├── Q3: {操作流程} - 具体操作步骤是什么？
├── Q4: {权限控制} - 谁可以执行此操作？
└── Q5: {边界条件} - 特殊情况如何处理？
```

### 分解示例

**原问题**："如果多人报工，如何查询每个人的报工记录？"

```
分解：
├── Q1: 多人报工能力 - 系统是否支持多人对同一工单报工？
├── Q2: 数据模型 - 报工记录如何存储？是否关联人员？
├── Q3: 查询功能 - 系统提供哪些查询条件？
├── Q4: 查询入口 - 查询界面在哪里？
└── Q5: 结果展示 - 查询结果如何展示人员信息？
```

## 证据收集技术

### 检索策略

#### 关键词扩展

根据问题类型扩展检索关键词：

| 问题类型 | 基础关键词 | 扩展关键词 |
|---------|----------|-----------|
| 功能类 | 功能名称 | 同义词、相关表名、页面名称 |
| 数据类 | 表名/字段名 | 实体名、业务术语、相关表 |
| 流程类 | 流程名称 | 状态名、动作名、角色名 |
| 权限类 | 角色名 | 功能名、权限矩阵、授权 |

#### 示例

```
问题：如何进行库存盘点？

基础关键词：库存盘点
扩展关键词：
- 盘点
- inventory
- stock_count
- 盘点单
- 盘点明细
- 盈亏处理
```

### 证据分级

根据证据的可靠性分级：

| 级别 | 来源 | 可靠性 | 示例 |
|-----|------|-------|------|
| L1 | 明确定义的功能/规则 | 最高 | PRD功能点、表结构 |
| L2 | 可推断的逻辑 | 高 | 从流程图可推断的步骤 |
| L3 | 原型/示例 | 中 | 原型设计中的字段/布局 |
| L4 | 会议纪要/讨论 | 低 | 决策背景，需谨慎使用 |

### 交叉验证

从多个来源验证同一事实：

```
验证项：报工记录是否包含人员信息

来源A（表结构）：
work_report.user_id INT NOT NULL
→ L1级证据，数据模型支持

来源B（功能描述）：
"每条报工记录关联操作人员"
→ L1级证据，功能定义支持

来源C（原型设计）：
报工列表包含"操作人"列
→ L3级证据，界面展示支持

交叉验证结果：三处一致，确认支持 ✅
```

## 缺陷识别技术

### 缺陷检测清单

在调查过程中，对照以下清单识别潜在缺陷：

#### 完整性检查

- [ ] 功能是否定义了输入？
- [ ] 功能是否定义了输出？
- [ ] 功能是否定义了操作者？
- [ ] 功能是否定义了前置条件？
- [ ] 功能是否定义了异常处理？

#### 一致性检查

- [ ] 功能描述与表结构是否一致？
- [ ] 状态值在不同文档中是否一致？
- [ ] 角色名称在不同章节是否一致？
- [ ] 业务规则在不同地方是否一致？

#### 可实现性检查

- [ ] 功能需要的字段是否在表中存在？
- [ ] 查询条件是否被索引支持？
- [ ] 业务流程是否有明确的起止点？
- [ ] 权限是否覆盖所有角色？

### 缺陷分类标准

#### 🔴 缺失 (Missing)

定义：文档中完全没有提及必要内容。

判断标准：
- 核心功能没有输入/输出定义
- 业务流程没有异常处理
- 数据表没有必要的字段

```
示例：
问题：订单取消后库存如何处理？
文档：未提及取消订单的库存回退逻辑
判定：🔴 缺失 - 核心业务场景未定义
```

#### 🔴 矛盾 (Conflict)

定义：不同文档或同一文档的不同位置描述相互矛盾。

判断标准：
- 同一字段在不同表中有不同约束
- 同一功能在不同章节有不同的流程描述
- 同一状态有不同的含义

```
示例：
文档A：用户名"不可重复"
文档B：用户名"允许重复，邮箱不可重复"
判定：🔴 矛盾 - 两处描述直接冲突
```

#### 🟡 模糊 (Ambiguous)

定义：描述不清晰，存在多种理解方式。

判断标准：
- 使用"相关人员"但未明确是谁
- 使用"适当处理"但未说明具体方式
- 使用"等"、"可能"等不确定词语

```
示例：
文档："订单提交后通知相关人员审批"
分析：未定义"相关人员"是谁
判定：🟡 模糊 - 执行主体不明确
```

#### 🟡 断层 (Gap)

定义：上层定义了，底层未支撑。

判断标准：
- 功能要展示X，但表结构无X字段
- 业务流程有某状态，但状态枚举中未定义
- 功能需要计算，但未定义计算规则

```
示例：
功能：显示"累计报工工时"
表结构：无累计字段，只有单次工时
分析：需要实时计算，但未定义计算逻辑
判定：🟡 断层 - 数据层支撑不完整
```

## 推理规范

### 推理类型

#### 直接推理

文档直接回答问题，无需额外推理。

```
问题：订单号格式是什么？
文档："订单号格式：年月日+4位序号，如202401150001"
结论：订单号为 YYYYMMDD + 4位序号
```

#### 演绎推理

从一般规则推导具体结论。

```
规则：所有订单需要审批
案例：这是一个订单
结论：这个订单需要审批
```

#### 归纳推理

从多个具体案例总结规律。

```
证据A：报工查询支持按工单筛选
证据B：报工查询支持按日期筛选
证据C：报工查询支持按班组筛选
归纳：系统支持多维度组合查询
```

### 推理标注

所有推理必须标注推理类型和置信度：

```
结论：用户可以查看自己的报工记录

推理过程：
1. 权限矩阵定义"普通员工可查看自己的数据"（直接证据）
2. 报工记录属于员工产生的数据（演绎）
3. 因此员工可以查看自己的报工记录

置信度：高（基于明确的权限规则）
```

### 推理限制

**禁止的推理**：
- ❌ 从其他项目经验推断
- ❌ 从"常见做法"推断
- ❌ 从未提及的内容推断"不支持"

**允许的推理**：
- ✅ 从明确定义演绎
- ✅ 从多个证据归纳
- ✅ 从上下文关联推断

## 调查过程记录

### 记录格式

在调查过程中，按以下格式记录：

```
## 调查记录

**问题**：{子问题}

**检索策略**：
- 关键词：{关键词列表}
- 目标文档：{文档列表}

**检索结果**：
- 📄 {文档名}：{匹配内容摘要}
- 📄 {文档名}：未找到相关内容

**证据评估**：
- 证据A：{内容}（L1级，直接证据）
- 证据B：{内容}（L2级，间接证据）

**结论**：
{基于证据的结论}

**缺口识别**：
{如果发现缺口，在此记录}
```

### 过程透明

调查过程应当透明可追溯：

1. 说明检索了哪些文档
2. 说明使用了什么关键词
3. 说明找到了什么证据
4. 说明推理过程是什么
5. 说明结论是什么

## 附录：常见问题类型调查指南

### 功能实现类

**调查路径**：
1. 功能点列表 → 确认功能存在
2. 表结构 → 确认数据支撑
3. 业务流程 → 确认操作步骤
4. 权限设计 → 确认操作权限

### 能力边界类

**调查路径**：
1. 功能点列表 → 确认支持范围
2. 非功能需求 → 确认性能限制
3. 约束条件 → 确认业务限制

### 业务场景类

**调查路径**：
1. 业务流程 → 确认正常流程
2. 异常处理 → 确认异常流程
3. 状态定义 → 确认状态流转
4. 表结构 → 确认数据一致性

### 数据查询类

**调查路径**：
1. 表结构 → 确认数据存储
2. 字段定义 → 确认数据粒度
3. 查询功能 → 确认查询条件
4. 原型设计 → 确认展示方式

### 权限流程类

**调查路径**：
1. 角色定义 → 确认角色列表
2. 权限矩阵 → 确认角色功能
3. 业务流程 → 确认操作节点
4. 审批规则 → 确认审批逻辑
