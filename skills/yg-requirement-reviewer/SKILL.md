# 需求文档审查与仿真验证

## 定位

这是一个**方法论框架**，适用于任何系统、任何领域的需求文档审查。支持多种需求文档类型：

| 文档类型 | 英文全称 | 核心侧重点 | 典型内容 |
|---------|---------|-----------|---------|
| **PRD** | Product Requirement Document | 产品功能与用户体验 | 功能列表、原型、交互流程 |
| **BRD** | Business Requirement Document | 商业价值与业务目标 | 市场分析、ROI、战略目标 |
| **SRS** | Software Requirement Specification | 技术规格与系统行为 | 功能/非功能需求、接口定义 |
| **TRD** | Technical Requirement Document | 技术实现方案 | 架构设计、技术选型、性能指标 |
| **其他** | 用户指定的需求内容 | 按内容自适应 | 功能点、流程、表设计等 |

本技能提供：审查流程、评估维度、判断标准、仿真方法和输出规范。具体的需求文档内容完全由用户提供，技能只负责按照标准化流程进行多维度审查，并输出结构化的审查报告。

## 核心理念

审查不是挑毛病，而是帮助需求文档变得更好。每一条发现都必须附带具体建议。评分和分级是手段，让用户快速定位最需要关注的问题才是目的。

---

## 模板资源

审查报告默认输出 **Markdown 格式**。如需 HTML 可视化报告（含 Mermaid 流程图和 Chart.js 图表），审查完成后会自动询问用户，用户同意后调用 `yg-visualize` 技能生成。


### 输入要求

用户提供需求文档（Markdown文件或文本内容），可以是以下任意类型：

- **PRD**（产品需求文档）
- **BRD**（商业需求文档）
- **SRS**（软件需求规格说明书）
- **TRD**（技术需求文档）
- **其他**（用户指定的需求设计文档或内容）

文档中建议包含以下部分的信息：

- 项目背景/概述
- 功能需求（功能点列表）
- 业务流程
- 数据表设计（DDL或表结构描述）

**智能适配**：技能会根据文档类型和内容自动调整审查重点：
- **PRD** → 侧重功能完整性、用户体验、交互逻辑
- **BRD** → 侧重商业目标、市场定位、ROI论证
- **SRS** → 侧重需求规格的精确性、可测试性、完整性
- **TRD** → 侧重技术可行性、架构合理性、性能指标
- **通用内容** → 自动识别并适配审查维度

如果文档中缺少上述某些部分，不要拒绝审查，而是：
1. 先审查已有的部分
2. 在报告中明确指出缺失的部分及其影响
3. 给出补充建议

### 规模感知

在开始审查前，先评估项目规模。这决定了审查的严格程度和建议的方向：

| 信号 | 小型项目 | 中型项目 | 大型项目 |
|------|---------|---------|---------|
| 功能点数量 | <10 | 10-30 | >30 |
| 表数量 | <8 | 8-25 | >25 |
| 用户角色 | 1-2 | 3-5 | >5 |
| 业务流程 | 简单线性 | 有分支 | 多系统交互 |

识别规模后，调整审查尺度——小项目不需要微服务架构，大项目不能缺少权限矩阵。

### 审查效率原则

对不同规模的需求文档，采用不同的审查深度：
- **小型文档**（功能点<10，表<8）：直接进行完整审查，但可跳过对不相关章节的逐项检查
- **中型文档**：完整执行所有阶段
- **大型文档**：增加角色-功能矩阵交叉检查、跨模块依赖分析

### 文档类型适配

不同类型的需求文档，审查侧重点有所不同：

| 文档类型 | 重点审查维度 | 可适当简化的维度 |
|---------|-------------|-----------------|
| PRD | 功能完整性、业务流程、仿真测试 | 技术架构细节 |
| BRD | 规模适配性、商业目标合理性 | 表结构细节、仿真测试 |
| SRS | 功能完整性、一致性校验、边界条件 | 规模适配性 |
| TRD | 规模适配性、可扩展性、性能指标 | 用户角色、权限设计 |
| 通用内容 | 自动识别并适配 | - |

## 审查流程

按以下六个阶段依次进行。

---

### 第一阶段：文档格式与结构审查

好的需求文档应该让任何团队成员都能快速找到需要的信息。这一步检查的是文档本身的"可用性"。

**结构完整性检查清单**：

- [ ] 项目概述（背景、目标、范围边界）
- [ ] 术语表 / 名词定义（如果涉及领域术语）
- [ ] 用户角色定义（如适用）
- [ ] 功能需求列表（带编号，可追溯）
- [ ] 业务流程描述（文字或流程图）
- [ ] 数据模型 / 表设计（如适用）
- [ ] 权限设计（角色-功能矩阵，如适用）
- [ ] 非功能性需求（性能、安全、可用性，如适用）
- [ ] 接口设计 / API 规划（如适用）

> 💡 **提示**：不同类型文档的关注点不同。BRD更关注商业目标，TRD更关注技术指标，PRD/SRS更关注功能细节。根据文档类型灵活调整检查重点。

**格式规范检查**：

- 功能点是否有唯一编号（便于追溯和引用）
- 描述是否清晰、无歧义（避免"等"、"可能"、"大概"等模糊词）
- 流程是否有明确的起止点和异常分支
- 表设计是否包含字段名、类型、约束、说明

**评分**：
- A（90-100）：结构完整，格式规范，信息充分
- B（70-89）：基本完整，有少量缺失可补充
- C（50-69）：缺失较多关键部分
- D（<50）：不具备进入开发的条件

对于缺失的部分，说明它的缺失会在后续哪个环节造成什么问题，而不只是列出"缺少XX"。

---

### 第二阶段：规模适配性审查

需求文档最大的陷阱之一是规模错配——小项目写了企业级架构，大项目用了MVP式的简化设计。

**过度设计信号**（对小型项目）：
- 功能点少于10个却有超过15张表
- 只有1-2个角色却设计了复杂的RBAC权限体系
- 数据量预期很小却设计了分库分表方案
- 简单业务却引入了工作流引擎、规则引擎等重型中间件
- 大量"预留字段"或"未来扩展"的设计

**设计不足信号**（对中大型项目）：
- 多角色系统却没有权限矩阵
- 涉及资金/库存变动却没有事务设计说明
- 多状态流转却没有状态机定义
- 高并发场景却没有并发控制策略
- 缺少数据备份/恢复策略

**评分**：
- A：规模匹配，设计既不过度也不不足
- B：略有偏差，调整成本低
- C：明显错配，需要较大调整
- D：严重错配，建议重新评估设计方向

---

### 第三阶段：功能点完整性与一致性校验

逐一检查每个功能点，验证它在需求文档内部的逻辑自洽性。

**对每个功能点进行以下检查**：

1. **数据支撑性**：该功能涉及的每一项数据，在表设计中是否都有字段承接？
2. **流程覆盖性**：该功能的操作流程是否在业务流程中有完整描述？包括正常流程和异常分支？
3. **权限匹配性**：该功能被哪些角色使用？权限设计中是否有对应的授权？
4. **依赖关系**：该功能是否依赖其他功能？被依赖的功能是否已定义？是否存在循环依赖？
5. **边界条件**：极端情况（空数据、最大量、并发操作）是否有说明？

**表结构内部一致性检查（冲突检测）**：

这是本技能的一个重要检查点，专门发现表设计内部的自我矛盾：

- **外键与字段矛盾**：某表有 `role_id` 字段（暗示一人一角色），同时又存在 `user_role` 关联表（暗示一人多角色）——两者只能存在一个
- **状态枚举矛盾**：表设计中的状态值（如 `status=3`）在功能描述中从未出现——要么是多余的死代码，要么是遗漏的功能
- **字段约束矛盾**：某字段声明为 NOT NULL 但没有默认值，而功能描述中没有该字段的必填场景
- **命名不一致**：同一业务概念在不同表中使用了不同的命名（如 `user_id` vs `buyer_id` vs `member_id`）

**交叉一致性检查**：

- 功能点描述中提到的状态值，是否和表设计中的枚举/字典一致？
- 流程中的角色名称，是否和权限矩阵中的角色名称一致？
- 功能点编号在不同章节引用时是否一致？
- 同一个业务概念在不同章节中是否使用了不同的名称？

**发现问题时标注类型**：
- **🔴 冲突**：两处描述相互矛盾（必须解决）
- **🟡 遗漏**：某处引用了未定义的内容（需要补充）
- **🟠 歧义**：描述可以有多种理解（需要澄清）

---

### 第四阶段：业务流程校验

业务流程是功能的骨架。这一步从流程视角检查需求文档的完整性。

**流程分析方法**：

对每个核心业务流程：

1. **画出状态流转图**：提取流程中涉及的所有状态，检查：
   - 每个状态是否有明确的进入条件和退出条件
   - 是否存在"死胡同"状态（进得去出不来）
   - 是否存在状态跳跃（从A直接到C，跳过了必要的B）
   - 终态是否明确
   - 是否有"幽灵状态"（表中有但流程中从未出现）

2. **角色参与分析**：标注每个流程节点由哪个角色操作：
   - 是否存在无人负责的节点
   - 是否存在职责冲突（同一节点需要两个互斥角色）

3. **异常路径覆盖**：
   - 操作失败时怎么办？
   - 超时未处理时怎么办？
   - 数据不一致时怎么办？
   - 审批驳回后是回到上一步还是回到起点？

4. **跨流程交互**：
   - 不同流程之间是否有数据依赖？
   - 一个流程的状态变更是否会影响另一个流程？

---

### 第五阶段：仿真模拟测试

这是本技能最核心的差异化能力。在没有真实代码和界面的情况下，仅以**表结构作为唯一事实来源**，对每个功能点进行逻辑性推理验证。

#### 仿真原理

表结构是系统的"骨架"——如果骨架无法支撑某个动作，那么无论代码写得多好，这个功能都无法正确实现。仿真测试就是在纸面上"执行"每个功能，看表结构是否经得起考验。

#### 仿真步骤

**Step 1：功能点提取与分类**

从需求文档中提取所有功能点，按操作类型分类：

| 类型 | 说明 | 示例 |
|------|------|------|
| C（Create） | 创建/新增数据 | 新建订单、注册用户 |
| R（Read） | 查询/展示数据 | 订单列表、统计报表 |
| U（Update） | 修改/状态变更 | 审批通过、编辑资料 |
| D（Delete） | 删除/作废数据 | 取消订单、删除草稿 |
| X（Complex） | 涉及多表联动的复杂操作 | 下单（扣库存+创建订单+生成支付） |

**Step 2：逐功能点仿真推演**

对每个功能点，模拟一次完整的操作过程。**每个失败案例必须包含"最小修复方案"**：

```
功能点：[功能名称]（编号：[XX]）
操作类型：[C/R/U/D/X]

--- 仿真过程 ---

1. 触发条件：[什么角色在什么场景下触发此功能]
2. 输入数据：[用户需要提供什么数据]
3. 涉及的表：[列出所有相关表]
4. 操作序列：
   a. [第一步操作：INSERT/UPDATE/SELECT 哪张表的哪些字段]
      → [操作结果：✅成功 / ❌字段不存在 / ⚠️逻辑不明]
   b. [第二步操作：...]
      → [操作结果]
   c. ...

--- 仿真结论 ---

判定：✅ 通过 / ⚠️ 存在风险 / ❌ 无法实现
原因：[具体说明]

💡 最小修复方案：[用一句话说明需要做什么最小的修改就能解决此问题]
```

**最小修复方案的格式要求**：
- 一句话说明，不超过30字
- 给出具体的修改方向（如"在XX表增加XX字段"，"新增XX表"）
- 如果当前信息不足以给出最小修复，明确说明"需要补充XX信息后才能确定"

**Step 3：关联性仿真**

单独的功能点可能各自没问题，但组合起来可能有问题。检查：

- **数据流转测试**：数据从创建到最终使用的完整链路是否通畅？
  例：用户下单 → 订单表写入 → 库存表更新 → 财务表记录 → 报表查询展示

- **并发冲突测试**：两个用户同时操作同一数据时，表结构是否能避免冲突？
  例：两人同时审批同一订单，状态字段如何保证一致性？

- **回环测试**：涉及状态流转的功能，是否能正确完成一个完整的生命周期？
  例：订单从创建→支付→发货→收货→完成→退款→关闭的全过程

**Step 4：可扩展性评估**

基于当前表结构，评估以下扩展场景的难度（仅评估合理的近期扩展，不做过远假设）：

- 新增一种用户角色，需要改多少张表？
- 新增一个类似的业务流程，需要新建多少张表？
- 数据量增长10倍后，现有索引策略是否还能支撑？

**可扩展性评分**：
- A（优秀）：扩展新功能只需加表加字段，不需要改现有结构
- B（良好）：小幅调整现有表即可扩展
- C（一般）：扩展需要较大的结构调整
- D（较差）：扩展几乎需要重新设计

## 审查原则

- **立场中立，面向解决方案**。审查者不是对立面，而是帮助需求文档作者发现盲点。每个问题都应该附带可操作的建议。
- **具体到行，精确到字段**。"权限设计有问题"没有价值。"功能F003（审批订单）要求'主管'角色操作，但权限矩阵中'主管'角色没有'订单审批'权限"才有价值。
- **展示因果链条**。不要只指出问题，要说明这个问题会在什么时候、以什么方式伤害到项目。
- **尊重项目规模**。内部3人使用的小工具不需要企业级设计；面向百万用户的系统不能缺少并发设计。
- **不编造，不假设**。只基于文档中实际写出的内容进行审查。信息不足时要求补充，不要脑补。
- **量化优于定性**。能打分就打分，能统计就统计。"功能覆盖率 85%（17/20 通过）"比"大部分功能都有覆盖"有用得多。
- **最小修复优于完美方案**。在仿真测试中，给出的修复建议应该是"最小改动就能解决问题"，而不是"重新设计整个系统"。
- **文档类型适配**。根据文档类型（PRD/BRD/SRS/TRD）调整审查重点和严格程度，不一把尺子量到底。
