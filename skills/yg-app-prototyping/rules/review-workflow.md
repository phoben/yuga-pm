# 审查流程规则

本文档定义代码审查与合并的完整流程。

---

## 审查流程图

```
PR 就绪
    │
    ▼
┌─────────────────────────────────────┐
│          审查者 Agent               │
│                                     │
│  1. 获取 PR 变更                   │
│  2. 执行审查检查:                  │
│     □ DRD 对齐检查                 │
│     □ 代码质量检查                 │
│     □ 组件规范检查                 │
│     □ 无障碍检查                   │
│  3. 生成审查报告                   │
└───────────────┬─────────────────────┘
                │
        ┌───────┴───────┐
        ▼               ▼
    通过           需要修改
        │               │
        ▼               ▼
  合并 PR       SendMessage(反馈)
        │               │
        ▼               ▼
  标记任务完成   Coder 修复并更新 PR
                        │
                        └──→ 重新审查
```

## 审查检查项

### 1. DRD 对齐检查

| 检查项 | 说明 |
|-------|------|
| 功能完整性 | 代码是否实现了 DRD 中定义的所有功能 |
| 页面结构 | 页面布局是否符合 DRD 设计 |
| 组件完整性 | 是否包含所有必要的 UI 元素 |
| 交互实现 | 用户交互是否按预期工作 |

### 2. 代码质量检查

| 检查项 | 说明 |
|-------|------|
| 命名规范 | 变量、函数、组件命名是否清晰一致 |
| 代码结构 | 逻辑是否清晰，模块划分是否合理 |
| 错误处理 | 是否有适当的错误处理和边界检查 |
| 类型定义 | TypeScript 类型是否完整准确 |

### 3. 组件规范检查

| 检查项 | 说明 |
|-------|------|
| UI 库模式 | 是否遵循所选 UI 库（shadcn/ui 等）的最佳实践 |
| 组件复用性 | 是否可以复用，是否过度耦合 |
| Props 定义 | 组件 Props 类型是否完整，是否有默认值 |
| 样式规范 | 样式是否符合设计令牌规范 |

### 4. 无障碍检查

| 检查项 | 说明 |
|-------|------|
| ARIA 标签 | 交互元素是否有正确的 aria 属性 |
| 键盘导航 | 是否支持键盘操作 |
| 颜色对比度 | 文字与背景对比度是否达标 |
| 语义化 HTML | 是否使用正确的语义化标签 |

### 5. 性能检查

| 检查项 | 说明 |
|-------|------|
| 渲染性能 | 是否存在不必要的重渲染 |
| 数据加载 | 数据加载策略是否合理 |
| 资源优化 | 图片、资源是否优化 |
| 代码分割 | 是否需要懒加载/代码分割 |

## 问题分类

| 级别 | 英文 | 说明 | 处理方式 |
|-----|------|------|---------|
| Critical | Critical | 必须修复的问题，影响功能或安全 | 阻止合并，必须修复后才能通过 |
| Important | Important | 建议修复的问题，影响可维护性或用户体验 | 可选择修复，记录在报告中 |
| Suggestion | Suggestion | 改进建议，代码优化方向 | 仅记录，不阻止合并 |

### Critical 问题示例

- 缺少关键功能实现
- 安全漏洞（XSS、注入等）
- 严重的无障碍问题
- 导致崩溃的错误

### Important 问题示例

- 缺少错误处理
- 不一致的命名
- 可以抽取为复用组件的重复代码
- 缺少必要的注释

### Suggestion 问题示例

- 可以使用更简洁的语法
- 建议使用特定库/工具
- 性能优化建议

## 合并流程

### Step 1: 确认审查通过

```bash
# 检查 PR 状态
gh pr view {pr-number}
```

确保所有 Critical 问题已修复，审查结果为通过。

### Step 2: 合并 PR

```bash
# Squash 合并（推荐）
gh pr merge {pr-number} --squash

# 或普通合并
gh pr merge {pr-number} --merge
```

### Step 3: 更新任务状态

使用 TaskUpdate 标记任务完成：

```javascript
TaskUpdate({
  taskId: "{task-id}",
  status: "completed",
  output: "PR #{pr-number} merged successfully"
})
```

### Step 4: 清理分支

```bash
# 删除远程分支（如果未自动删除）
git push origin --delete feature/page-{name}

# 删除本地分支
git branch -d feature/page-{name}
```

## 审查反馈流程

### 审查不通过时

```javascript
// 发送反馈给 Coder Agent
SendMessage({
  to: "coder-agent",
  message: `
## 审查反馈

PR #{pr-number} 需要修改：

### Critical 问题
1. {问题描述}

### Important 问题
1. {问题描述}

请修复后更新 PR。
  `,
  summary: "审查需要修改"
})
```

### Coder Agent 修复后

1. Coder Agent 收到反馈
2. 修复问题并提交新代码
3. 推送更新到现有 PR
4. 通知 Reviewer Agent 重新审查

### 重新审查

```bash
# 获取最新变更
gh pr diff {pr-number}

# 重新执行审查流程
```

---

## 反模式

- ❌ **Coder Agent 自我审查** - 这会造成利益冲突
- ❌ **跳过审查直接合并** - 破坏质量控制
- ❌ **忽略无障碍问题** - 影响用户体验和法律合规
- ❌ **过于宽松的审查** - 降低代码质量
- ❌ **忽略合并冲突** - 可能导致数据丢失
- ❌ **无限审查循环** - 设置最多 3 次重新审查

---

## 审查限制

| 限制项 | 值 | 说明 |
|-------|-----|------|
| 最大重新审查次数 | 3 | 超过 3 次需要人工介入 |
| 审查超时 | 10 分钟 | 超时后标记为需要人工审查 |
| PR 变更文件数上限 | 50 | 超过建议拆分 PR |

---

## 最佳实践

1. **及时反馈** - 审查完成后立即发送反馈
2. **明确优先级** - 区分 Critical/Important/Suggestion
3. **提供解决方案** - 不只指出问题，还提供修复建议
4. **保持一致性** - 使用统一的审查标准和模板
5. **文档化** - 所有审查结果保存为报告
