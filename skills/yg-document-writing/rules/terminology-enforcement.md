# 术语强制规则

## 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│ 所有SubAgent必须使用大纲中定义的术语，不得创造新术语           │
│                                                              │
│ ✅ 使用术语表定义的术语                                       │
│ ❌ 使用同义词或自创术语                                       │
│ ❌ 对同一概念使用不同称呼                                     │
└─────────────────────────────────────────────────────────────┘
```

## 术语表结构

在大纲阶段生成术语表：

```markdown
## 术语表

| 术语 | 定义 | 英文（如需） |
|------|------|--------------|
| 用户 | 系统的最终使用者 | User |
| 管理员 | 拥有系统管理权限的用户 | Administrator |
| 订单 | 用户提交的购买请求 | Order |
| 商品 | 系统中可购买的物品 | Product |
```

## 术语一致性检查

### 检查时机

1. **SubAgent编写章节时**：注入术语表，要求强制使用
2. **质量检查时**：扫描全文检查术语一致性

### 检查规则

| 规则 | 说明 |
|------|------|
| 术语存在性 | 文档中使用的术语必须在术语表中定义 |
| 术语唯一性 | 同一概念只能使用一个术语 |
| 定义完整性 | 术语表中的术语必须有明确定义 |

### 常见术语冲突示例

| 冲突类型 | 错误示例 | 正确做法 |
|----------|----------|----------|
| 同义多词 | "用户"、"使用者"、"客户"混用 | 统一使用"用户" |
| 一词多义 | "账户"既指用户账户又指银行账户 | 分别定义"用户账户"和"银行账户" |
| 中英混用 | "User"和"用户"混用 | 统一使用中文"用户" |

## SubAgent提示词模板

在注入上下文时，强制包含术语表：

```
## 术语表（强制使用）

以下是本文档定义的标准术语，你必须使用这些术语：

| 术语 | 定义 |
|------|------|
| [术语1] | [定义1] |
| [术语2] | [定义2] |

**禁止事项：**
- ❌ 使用同义词替代（如用"客户"代替"用户"）
- ❌ 创造新术语
- ❌ 中英文混用
```

## 术语扩展机制

### 新增术语

当SubAgent发现需要使用新术语时：

1. **不直接使用新术语**
2. **向主Agent报告**：发现新概念需要定义术语
3. **主Agent决定**：
   - 添加到术语表
   - 使用现有术语

### 术语冲突处理

当发现术语使用不一致时：

```
⚠️ 术语一致性警告

章节 3.1 中使用了术语"客户"，但术语表中定义为"用户"。

建议修正：将"客户"替换为"用户"
```

## 术语表生成指南

### 从原始需求中提取术语

1. 识别需求中的关键概念
2. 确定每个概念的标准术语
3. 为每个术语编写明确定义
4. 去除同义词，保留唯一术语

### 术语命名原则

| 原则 | 说明 |
|------|------|
| 简洁性 | 术语应简短易记 |
| 准确性 | 术语应准确反映概念本质 |
| 无歧义 | 术语不应有多种解释 |
| 一致性 | 与行业惯例保持一致 |

## 术语表示例

```markdown
## 术语表

| 术语 | 定义 |
|------|------|
| 用户 | 在系统中注册并使用系统功能的个人或组织 |
| 管理员 | 具有系统管理权限的特殊用户 |
| 商品 | 系统中可供用户浏览和购买的物品或服务 |
| 订单 | 用户提交的购买请求，包含商品、数量、金额等信息 |
| 支付 | 用户为完成订单而进行的资金转移行为 |
| 退款 | 将已支付金额返还给用户的操作 |
| 库存 | 商品的可用数量 |
| SKU | 库存单位，商品的最小管理单元 |
```
