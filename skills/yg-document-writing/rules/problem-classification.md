# 问题分级标准

## 三级分类体系

| 级别 | 定义 | 示例 | 处理方式 |
|------|------|------|----------|
| **轻微** | 格式问题，不影响内容理解 | 编号错误、标点不规范 | 自动修复 |
| **一般** | 内容问题，但不阻断交付 | 描述模糊、引用不准确 | 自动修复 |
| **严重** | 结构性或逻辑问题 | 需求遗漏、过度设计、矛盾描述 | 用户确认后修复 |

## 轻微问题判定标准

| 问题类型 | 判定条件 | 修复方式 |
|----------|----------|----------|
| 编号错误 | 章节编号不连续或格式错误 | 自动修正编号 |
| 标点不规范 | 中英文标点混用、缺少标点 | 自动修正标点 |
| 格式不统一 | 同类内容格式不一致 | 自动统一格式 |
| 拼写错误 | 明显的拼写或打字错误 | 自动修正 |

### 轻微问题示例

```
问题：章节编号从 2.1 跳到 2.3，缺少 2.2
级别：轻微
修复：重新编号为连续序列
```

## 一般问题判定标准

| 问题类型 | 判定条件 | 修复方式 |
|----------|----------|----------|
| 描述模糊 | 需求描述不够具体，但可推断 | 补充具体描述 |
| 引用不准确 | 章节引用编号有误但内容正确 | 修正引用编号 |
| 术语不一致 | 同一概念使用不同术语 | 统一为术语表定义 |
| 信息不完整 | 缺少非关键信息 | 补充必要信息 |

### 一般问题示例

```
问题：功能描述中使用"用户"，但术语表中定义为"系统使用者"
级别：一般
修复：统一使用"系统使用者"
```

## 严重问题判定标准

| 问题类型 | 判定条件 | 处理方式 |
|----------|----------|----------|
| 需求遗漏 | 原始需求未在文档中体现 | 用户确认后补充 |
| 过度设计 | 超出原始需求范围的设计 | 用户确认后删除或保留 |
| 矛盾描述 | 不同章节内容相互矛盾 | 用户确认后修正 |
| 逻辑错误 | 业务流程或数据流逻辑错误 | 用户确认后修正 |
| 关键信息缺失 | 缺少核心功能描述或数据定义 | 用户确认后补充 |

### 严重问题示例

```
问题：原始需求 REQ-005"数据导出"在文档中未找到对应章节
级别：严重
处理：上报用户确认是否遗漏

问题：文档描述了"用户积分系统"，但原始需求中未提及
级别：严重
处理：上报用户确认是否过度设计
```

## 问题分级流程

```
发现问题时
    │
    ├─ 是否影响需求覆盖？
    │     ├─ 是 → 严重
    │     └─ 否 → 继续
    │
    ├─ 是否存在逻辑矛盾？
    │     ├─ 是 → 严重
    │     └─ 否 → 继续
    │
    ├─ 是否超出需求边界？
    │     ├─ 是 → 严重
    │     └─ 否 → 继续
    │
    ├─ 是否影响内容理解？
    │     ├─ 是 → 一般
    │     └─ 否 → 轻微
```

## 迭代修复策略

| 问题组合 | 迭代策略 |
|----------|----------|
| 全部轻微/一般 | 自动修复后重新检查，最多3次 |
| 存在严重问题 | 先上报用户确认，再修复 |
| 3次迭代后仍有问题 | 上报用户决定 |
